02. EXECUTIVE ARCHITECTURE STATEMENT
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

CarbonPulse is a full-stack Progressive Web App built on a layered architecture:
  - React 18 + TypeScript frontend (Vite, PWA-enabled)
  - Node.js + Express + TypeScript backend (REST API)
  - PostgreSQL primary store + Redis cache/sessions
  - Firebase Auth + Firestore for real-time sync
  - Six distinct Google platform services (not decoration — functionally load-bearing)
  - Comprehensive test suite: unit + integration + E2E + accessibility + performance
  - OWASP-aligned security middleware stack

The architecture separates CONCERNS CLEANLY:
  1. Carbon data ingestion (activities logged by user)
  2. Emission calculation engine (rule-based + regional factors)
  3. AI insight generation (Google Gemini API, async)
  4. Visualization layer (Recharts + custom D3 heatmap)
  5. Notification system (Firebase Cloud Messaging)
  6. Community benchmarking (anonymized aggregate data)

This is NOT a CRUD app with a carbon table. It is a data intelligence platform
with carbon as its domain — the same way Robinhood is not a database of stocks.

WHAT EVERY OTHER TEAM WILL BUILD:
  - Input form → Calculate CO2 → Show number → Done.
  - One Google Maps call for a route.
  - No tests, no auth, no persistence beyond localStorage.
  - A pie chart from a tutorial.

WHAT CARBONPULSE DOES DIFFERENTLY:

  [1] PRECISION CARBON ENGINE (Not Estimates — Calibrated Factors)
      Uses IPCC AR6 2023 emission factors, region-specific electricity grids,
      and dietary lifecycle assessment data. Not "car = 200g/km". Correct answer:
      diesel SUV India grid electricity = 198.4g CO2e/km (factored).

  [2] GOOGLE MAPS ROUTE INTELLIGENCE
      User doesn't input "I drove 20km". They click "Record Trip" and pick
      two locations via Google Places Autocomplete. Maps Routes API returns
      actual distance + mode → fed directly into the emission calculator.
      Transport mode switching shows live CO2 delta (drive vs train vs cycle).

  [3] GEMINI AI PERSONALIZED INSIGHT ENGINE (Not Generic Tips)
      Every Monday, Gemini generates a 3-sentence insight SPECIFIC to that
      user's pattern: "Your food emissions spiked 34% this week — three
      chicken meals on Tuesday contributed 2.1 kgCO2e. Swapping to legumes
      twice a week saves ≈47 kg annually, equivalent to 1 domestic flight."
      This is the killer feature. Nobody else will have personalized AI prose.

  [4] COMMUNITY BENCHMARKING (Firebase Realtime)
      Users see anonymized percentile: "Your carbon is lower than 67% of
      CarbonPulse users in Rajasthan." This is Firebase Realtime Database
      doing live aggregate calculations — not a static infographic.

  [5] STREAK + CARBON BUDGET SYSTEM (Gamification)
      Weekly carbon budget (default: 100 kgCO2e). Daily streak for staying
      under daily target. Badge system: "3-day green streak", "Flight-free
      month", "Vegan week". Streak data stored in Redis for fast reads.

  [6] PWA WITH OFFLINE ACTIVITY LOGGING
      Service worker caches the app shell. Activities logged offline get
      queued in IndexedDB and sync when connection returns. This is real
      PWA engineering, not just a manifest.json.

  [7] CARBON OFFSET MARKETPLACE RECOMMENDATIONS
      After 30 days, suggest verified offset projects (Gold Standard,
      Verra VCS) based on total footprint. Data fetched via web API.

ARCHITECTURAL DECISIONS (ADRs):

ADR-001: Monorepo with pnpm workspaces
  Reason: Shared TypeScript types between client and server prevent
  contract drift. Single CI pipeline. Shared ESLint config.
  Alternative rejected: Separate repos. Too much overhead, type drift risk.

ADR-002: Prisma ORM over raw SQL
  Reason: Type-safe DB access eliminates SQL injection class entirely
  (not just mitigated — structurally impossible). Migration history.
  Alternative rejected: Knex. Less type safety, more boilerplate.

ADR-003: Zod for runtime validation at API boundary
  Reason: TypeScript types are compile-time only. Zod validates incoming
  JSON at runtime, generates meaningful 400 errors, and the schema is
  the single source of truth for API contracts.
  Alternative rejected: Joi. Less TypeScript integration.

ADR-004: Firebase for real-time features, PostgreSQL for persistent data
  Reason: Firestore's real-time listeners handle community benchmarking
  and streak updates without polling. PostgreSQL handles complex queries,
  joins, and the emission factor reference data.

ADR-005: Zustand over Redux for client state
  Reason: This app's state is not highly interconnected. Zustand is
  ~8x less boilerplate, equally predictable, and tree-shakeable.
  Alternative rejected: Redux Toolkit. Overkill for this scope.

ADR-006: Workbox for Service Worker (PWA)
  Reason: Google-maintained, production-tested, handles cache strategies
  (NetworkFirst for API, CacheFirst for static assets), background sync.

07. GOOGLE SERVICES INTEGRATION PLAN (Target: 92/100)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The previous 75/100 means Google Services were cosmetic. This plan makes each
one ARCHITECTURALLY ESSENTIAL — remove any one and a core feature breaks.

  [1] FIREBASE AUTHENTICATION
      Integration depth: Deep (auth is the entire security boundary)
      Usage: Google Sign-In + Email/Password. Firebase Admin SDK verifies
      JWT tokens in every API middleware call. Token refresh handled client-side
      via Firebase SDK (automatic). User UID is foreign key in PostgreSQL.
      File: packages/server/src/middleware/auth.middleware.ts
      Test: packages/server/tests/integration/auth.test.ts

  [2] GOOGLE MAPS JAVASCRIPT API + PLACES API
      Integration depth: Deep (Transport logging doesn't work without it)
      Usage: TransportPicker.tsx renders an embedded map. User types origin
      in a Places Autocomplete input, destination in a second input.
      Places API returns lat/lng pairs. Polyline shown on map.
      File: packages/client/src/components/activities/TransportPicker.tsx
      File: packages/client/src/services/maps.service.ts

  [3] GOOGLE ROUTES API (Distance Matrix)
      Integration depth: Deep (emission calculation accuracy depends on it)
      Usage: Backend routes service calls Routes API with origin + destination
      lat/lng to get precise route distance in meters. This feeds directly into
      the CarbonCalcService. Transport mode is also returned for validation.
      Why server-side: Protect API key. Allows server-side caching of routes.
      File: packages/server/src/services/maps.service.ts
      Example: POST /api/activities/transport-preview
               Body: { origin: {lat, lng}, destination: {lat, lng}, mode: 'DRIVE' }
               Response: { distance_m: 15420, estimated_co2_g: 3084.0 }

  [4] GOOGLE GEMINI 1.5 FLASH API
      Integration depth: Deep (the entire Insights page depends on this)
      Usage: Once per week (cron at Sunday 23:59), Gemini generates a
      personalized insight for each active user. The prompt is constructed
      from the user's last 7 days of activity data, their category breakdown,
      their trend vs previous week, and their stated goals. Gemini returns
      structured JSON with: insight_text, primary_driver, suggested_action,
      projected_annual_saving_kg. This JSON is stored in PostgreSQL and
      served to the Insights page. The response is cached in Redis for 24h
      to avoid redundant API calls on page refresh.
      File: packages/server/src/services/gemini.service.ts
      Prompt template: See CARBON CALCULATION ENGINE section.

  [5] FIREBASE FIRESTORE (Real-time Community Benchmarking)
      Integration depth: Medium-Deep (Community page is real-time via Firestore)
      Usage: A Firestore collection 'community_stats' stores anonymized, 
      aggregated daily totals per region (India/Rajasthan/Global). Client
      uses onSnapshot() to receive live updates. User sees their percentile
      rank update in real-time as others log activities. No user PII in
      Firestore — only aggregated regional totals and the current user's
      position in the distribution.
      File: packages/client/src/hooks/useStreak.ts (Firestore listener)
      Collection: community_stats/{region}/daily/{YYYY-MM-DD}

  [6] FIREBASE CLOUD MESSAGING (Push Notifications)
      Integration depth: Medium (engagement layer for the insights feature)
      Usage: When a weekly Gemini insight is generated, FCM sends a push
      notification to the user's registered devices: "Your weekly carbon
      report is ready. You saved 12% vs last week!" Service worker handles
      the push event and shows notification even when app is closed.
      File: packages/server/src/services/fcm.service.ts
      File: packages/client/public/firebase-messaging-sw.js

  [7] GOOGLE ANALYTICS 4 (Behavioral Intelligence)
      Integration depth: Light (but correctly implemented via event schema)
      Usage: GA4 with custom events for feature engagement tracking.
      Events: activity_logged (category, co2_grams), insight_viewed (week_no),
      goal_set (target_kg), offset_clicked (project_id), streak_achieved (days).
      File: packages/client/src/services/analytics.service.ts
      Note: No PII in GA4. Only aggregated usage events.

  TRANSPORT (gCO2e per passenger-km):
    Petrol car (India avg)     : 192 g/km
    Diesel car (India avg)     : 171 g/km
    Motorcycle (India avg)     : 103 g/km
    Bus (Indian cities)        :  89 g/km
    Metro/Suburban Rail        :  41 g/km
    Domestic Flight (India)    : 255 g/km (includes RFI multiplier 1.9x)
    International Flight (eco) : 195 g/km
    Bicycle / Walking          :   0 g/km

  FOOD (gCO2e per kg of food):
    Beef (feedlot)             : 99,500 g/kg (99.5 kgCO2e/kg)
    Lamb                       : 39,200 g/kg
    Pork                       : 12,100 g/kg
    Chicken                    :  9,870 g/kg
    Fish (wild caught)         :  5,100 g/kg
    Dairy (milk, 1L)           :  3,200 g/kg
    Eggs (per dozen)           :  4,500 g/kg
    Rice (Indian paddy)        :  4,450 g/kg  (methane-intensive)
    Legumes / Pulses           :  2,000 g/kg
    Vegetables (local, India)  :    980 g/kg
    Vegetables (imported)      :  2,100 g/kg

  HOME ENERGY (gCO2e per unit):
    Electricity (India avg)    : 820 g/kWh (CEA grid intensity 2023)
    Electricity (Rajasthan)    : 890 g/kWh (coal-heavy state grid)
    Piped Natural Gas          : 2,204 g/m3
    LPG cooking gas (1 cylinder 14.2kg) : 44,100 gCO2e

  SHOPPING (gCO2e per item category):
    Cotton t-shirt (fast fashion) : 10,000 g
    Jeans (cotton denim)          : 33,000 g
    Smartphone (new)              : 70,000 g
    Laptop (new)                  : 350,000 g
    Car purchase (amortised/yr)   : 8,500,000 g/year

  CALCULATION FORMULA:
    TotalCO2e = Σ(activity_quantity × emission_factor × regional_multiplier)

    regional_multiplier for electricity defaults to 1.0 for national average.
    Set to 1.085 for Rajasthan, 0.89 for Maharashtra (more renewables), etc.

model User {
  id                String      @id @default(cuid())
  firebaseUid       String      @unique           // Firebase Auth UID
  email             String      @unique
  displayName       String?
  avatarUrl         String?
  region            String      @default("IN")    // ISO 3166 for regional factors
  weeklyBudgetKg    Float       @default(100.0)
  createdAt         DateTime    @default(now())
  updatedAt         DateTime    @updatedAt
  activities        Activity[]
  goals             Goal[]
  insights          Insight[]
  fcmTokens         FcmToken[]
}

model Activity {
  id              String      @id @default(cuid())
  userId          String
  user            User        @relation(fields: [userId], references: [id])
  category        Category
  subcategory     String      // e.g. "petrol_car", "chicken", "electricity"
  quantity        Float       // km, kg, kWh, etc.
  unit            String      // "km", "kg", "kWh"
  co2Grams        Float       // calculated value, stored for fast aggregation
  description     String?     // optional user note
  loggedAt        DateTime    @default(now())
  originLat       Float?      // Transport: origin coordinates
  originLng       Float?
  destLat         Float?      // Transport: destination coordinates
  destLng         Float?
  createdAt       DateTime    @default(now())

  @@index([userId, loggedAt])          // covers 90% of queries
  @@index([userId, category, loggedAt])
}

model EmissionFactor {
  id              String   @id @default(cuid())
  category        Category
  subcategory     String   @unique
  factorGPerUnit  Float    // gCO2e per unit
  unit            String
  region          String   @default("IN")
  source          String   // "IPCC AR6 2023", "CEA 2023"
  validFrom       DateTime
  metadata        Json?    // additional context (transport mode, diet type, etc.)

  @@index([category, subcategory, region])
}

model Goal {
  id              String      @id @default(cuid())
  userId          String
  user            User        @relation(fields: [userId], references: [id])
  targetKgPerWeek Float
  category        Category?   // null = overall goal
  startDate       DateTime
  endDate         DateTime?
  isActive        Boolean     @default(true)
  createdAt       DateTime    @default(now())
}

model Insight {
  id                    String   @id @default(cuid())
  userId                String
  user                  User     @relation(fields: [userId], references: [id])
  weekStartDate         DateTime
  insightText           String
  primaryDriver         String
  suggestedAction       String
  projectedSavingKgYear Float
  totalCo2KgForWeek     Float
  deltaFromPrevWeek     Float    // percentage
  rawGeminiResponse     Json?    // store for debugging
  createdAt             DateTime @default(now())

  @@unique([userId, weekStartDate])
}

model FcmToken {
  id        String   @id @default(cuid())
  userId    String
  user      User     @relation(fields: [userId], references: [id])
  token     String   @unique
  createdAt DateTime @default(now())
}

enum Category {
  TRANSPORT
  FOOD
  ENERGY
  SHOPPING
  WASTE
}

10. REST API SPECIFICATION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

BASE URL: /api/v1
AUTH: All endpoints require Authorization: Bearer <firebase_jwt>
      except /auth/verify-token and public /health.

AUTH ENDPOINTS:
  POST   /auth/verify-token        Verify Firebase JWT, create user if first login
  POST   /auth/refresh             Refresh session (Redis-backed)
  DELETE /auth/logout              Invalidate session token in Redis

USER ENDPOINTS:
  GET    /users/me                 Get current user profile
  PATCH  /users/me                 Update profile (region, budget, display name)
  DELETE /users/me                 Account deletion (GDPR - purge all data)

ACTIVITY ENDPOINTS:
  GET    /activities               List user activities (paginated, filterable)
         ?page=1&limit=20&category=TRANSPORT&from=2026-06-01&to=2026-06-30
  POST   /activities               Log new activity
         Body: { category, subcategory, quantity, unit, loggedAt?, description? }
         Response: { ...activity, co2Grams (calculated server-side) }
  GET    /activities/:id           Get single activity
  PATCH  /activities/:id           Update activity
  DELETE /activities/:id           Delete activity
  POST   /activities/transport-preview  Preview CO2 for transport route
         Body: { originLat, originLng, destLat, destLng, mode }
         Response: { distanceMeters, co2Grams, durationSeconds }

INSIGHTS ENDPOINTS:
  GET    /insights                 List all user insights (newest first)
  GET    /insights/latest          Get this week's insight (or trigger generation)
  POST   /insights/generate        Manually trigger Gemini insight generation
  GET    /insights/weekly-summary  Get week-by-week summary for chart

GOALS ENDPOINTS:
  GET    /goals                    List user goals

11. SECURITY ARCHITECTURE (OWASP TOP 10 2021 ALIGNED)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

A01 — BROKEN ACCESS CONTROL:
  - Every route requires auth middleware (no accidental public routes)
  - Row-level security: users can only access their own activities
    (enforced via userId === req.user.uid in all DB queries, not just routes)
  - Admin routes non-existent in this app (no privilege escalation surface)

A02 — CRYPTOGRAPHIC FAILURES:
  - HTTPS enforced via HSTS header (Helmet: max-age=31536000; includeSubDomains)
  - Passwords hashed with Argon2id (not bcrypt — OWASP 2024 recommendation)
  - Firebase JWTs signed with RS256 (asymmetric, unguessable)
  - No sensitive data in localStorage (tokens stored in memory or httpOnly cookies)
  - Google API keys scoped: Maps key restricted to HTTP referrer (domain)
  - Gemini/Routes keys backend-only, never in frontend bundle

A03 — INJECTION:
  - Prisma ORM: parameterized queries by design — SQL injection structurally impossible
  - Input sanitization via DOMPurify on all string inputs before DB write
  - Zod validation rejects unexpected fields (strict schemas)
  - No eval(), no dynamic SQL concatenation anywhere

A05 — SECURITY MISCONFIGURATION:
  - Helmet.js: CSP, X-Content-Type-Options, X-Frame-Options: DENY,
    Referrer-Policy: strict-origin-when-cross-origin
  - CSP: script-src 'self' *.googleapis.com *.firebaseapp.com; no unsafe-inline
  - CORS: explicit whitelist of allowed origins (not "*")
  - Error messages: production errors never expose stack traces
    (only logged server-side, generic message to client)
  - Environment validation at startup: app refuses to start if required
    env vars are missing (fail-fast, no partial configuration)

A07 — IDENTIFICATION & AUTHENTICATION FAILURES:
  - Rate limiting: 5 failed auth attempts per 15 minutes per IP (Redis counter)
  - Firebase handles brute force protection on the auth side
  - Session invalidation on logout (Redis blacklist of revoked token JTIs)
  - Token expiry: Firebase ID tokens expire after 1 hour (short-lived by design)

A08 — SOFTWARE AND DATA INTEGRITY FAILURES:
  - pnpm lockfile committed, no floating version ranges in production deps
  - Subresource Integrity not applicable (self-hosted frontend on Vercel)
  - Dependabot configured for weekly security alerts

ADDITIONAL:
  - All API responses include: X-Request-ID (trace ID for debugging)
  - Log all auth events (login, logout, failed attempts) to Winston
  - No PII in logs (email hashed/masked in log output)
  - Rate limiting: 100 requests/15min general, 20/15min for POST endpoints
  - File upload: not supported in this app (no file upload attack surface)

APPENDIX: KEY MANTRAS FOR EXECUTION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. "TESTS FIRST, ALWAYS."
   Create the /tests folder and at least one test file before writing
   any feature code. It is psychologically easier to maintain than to
   bolt on later. The zero we scored is 100% avoidable.

2. "GOOGLE SERVICES ARE ARCHITECTURALLY LOAD-BEARING, OR THEY DON'T COUNT."
   If you can remove a Google service and the app still works identically,
   it will score like a decoration. Every service must be on the critical path.

3. "NEVER EXPOSE BACKEND API KEYS TO THE FRONTEND BUNDLE."
   Routes API key, Gemini API key — server-only. The evaluator WILL check
   if these are in the client bundle (it's a security check, not just Google policy).

4. "ACCESSIBILITY IS NOT A CHECKLIST. IT IS A MINDSET."
   Write aria-label on charts the moment you write the chart.
   Associate error messages with inputs the moment you write the input.
   Run axe-core in your component test the moment you write the component.
   Retrofitting is 3x harder and always incomplete.

5. "PERSONALIZATION IS THE DIFFERENTIATOR."
   Every team will show a total CO2 number. Only CarbonPulse will say:
   "Your rice consumption (4 servings this week) added 17% to your food
   emissions. Reducing by 2 servings saves ~8 kg CO2e monthly."
   That specificity — powered by real IPCC factors + Gemini — is what
   makes this a 95+ app, not a 75 app.
